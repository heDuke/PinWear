# PinWear Agent 开发规范

本文档是 PinWear 项目的唯一软件开发规范，适用于 Codex、Claude Code、Gemini CLI 及其他 AI Coding Agent。除非用户明确要求，后续任务不得修改本文件。

## 工作原则

- 先理解现状，再提出方案，最后执行变更。
- 保持最小修改范围，避免与当前任务无关的重构。
- 优先复用现有技术栈、组件和目录结构。
- 所有结论必须以代码、配置、测试或用户明确说明为依据。
- 不确定的信息统一标记为“【待确认】”，不得自行补全业务事实。
- 不得把技术建议当成已确认需求。

## 开发流程

1. 阅读本文件和 `docs/` 目录。
2. 检查 Git 状态、当前分支和相关文件。
3. 明确需求范围、验收条件和不在范围内的内容。
4. 分析现状并记录风险或待确认事项。
5. 设计最小可行方案，必要时先请求用户确认。
6. 实施变更并同步相关文档。
7. 执行适合本次改动的自检、测试和构建。
8. 汇报修改内容、验证结果、未解决问题和下一步建议。

## 需求分析流程

- 明确用户目标、输入、输出、触发条件和验收标准。
- 区分“必须实现”“建议实现”和“暂不处理”。
- 检查需求是否涉及业务规则、权限、数据安全、兼容性或发布流程。
- 缺少关键业务信息时标记“【待确认】”，不得猜测。
- 先确认需求边界，再确定代码改动范围。

## 方案设计流程

- 先记录当前实现，再说明拟议实现。
- 优先选择与当前架构一致的方案。
- 说明状态流、数据流、错误处理、测试策略和回滚方式。
- 对会影响架构、依赖、数据格式、公开 API 或发布方式的方案，先请求用户确认。
- 方案必须包含明确的非目标，防止需求扩散。

## 用户确认流程

以下情况必须在执行前确认：

- 需求存在多种会产生明显产品差异的解释。
- 需要修改 Gradle、依赖、Manifest、包名或项目架构。
- 需要删除、迁移或覆盖用户数据。
- 需要新增外部服务、权限、密钥或付费资源。
- 需要进行超出当前任务范围的重构。

若可以安全地采用默认方案，应在汇报中说明假设；不能安全默认时停止并询问用户。

## Todo 管理流程

- 任务计划维护在 `docs/DEVELOPMENT_PLAN.md`。
- 使用 Markdown Checkbox：未完成为 `- [ ]`，完成为 `- [x]`。
- 每完成一个开发任务，必须同步更新开发计划和变更日志。
- Todo 必须可执行、可验证，避免使用“继续优化”等模糊描述。
- 被阻塞的 Todo 记录阻塞原因和待确认事项。

## 开发执行流程

- 开始前再次确认 Git 工作区，保护用户已有改动。
- 只修改完成当前任务所需的文件。
- 业务代码变更应配套测试或说明无法测试的原因。
- 不得为了通过检查而删除测试、降低质量阈值或隐藏错误。
- 不得修改生成文件、截图基线或本地 IDE 文件，除非任务明确要求。

## 自检流程

- 检查格式、编译错误、资源引用、字符串资源和可访问性。
- 检查不同 Wear OS 屏幕尺寸、圆形屏幕和字体缩放场景。
- 检查加载、成功、空数据、错误和恢复状态（若任务涉及）。
- 检查新增交互是否有自动化测试。
- 检查 `git diff`、`git status`，确保没有意外文件。
- 在汇报中区分“已验证”“未验证”和“环境阻塞”。

## Build 流程

- 默认使用项目自带 Gradle Wrapper，不使用系统 Gradle 替代。
- 常规验证优先执行 `./gradlew build lintDebug`。
- UI 截图相关变更执行对应的 Roborazzi 测试；只有在视觉变更经确认后才更新基线。
- 发布构建使用项目定义的 Release 流程，不在本地提交密钥。
- 当前 CI 对 Spotless 与 Gradle Configuration Cache 的兼容问题通过工作流参数规避；不要在无关任务中改变该策略。
- 构建失败时先定位首个失败任务和根因，不以最后一条错误作为唯一依据。

## Bug 修复流程

1. 复现问题并记录环境、输入和实际结果。
2. 定位根因，区分代码问题、配置问题、环境问题和外部服务问题。
3. 编写能覆盖回归场景的测试或验证步骤。
4. 采用最小修复，不顺便重构无关代码。
5. 运行回归验证并记录结果。
6. 更新 `docs/DEVELOPMENT_PLAN.md`、`docs/CHANGELOG.md`（如任务已完成）。

## 文档维护流程

- `DEVELOPMENT_PLAN.md`：阶段、Todo、完成项、风险和待确认事项。
- `ARCHITECTURE.md`：实际架构与已确认的架构约束。
- `API.md`：接口、数据契约和错误约定；没有 API 时明确记录“当前不存在”。
- `UI_GUIDELINES.md`：Wear OS、Compose 和 Material 3 的界面规范。
- `TEST_PLAN.md`：测试范围、策略、设备矩阵和验收标准。
- `DECISIONS.md`：重要技术决策及其背景。
- `CHANGELOG.md`：按日期记录项目变化。
- 业务、架构、API 或测试策略发生变化时，必须同步对应文档。

## Git Commit 规范

- Commit message 使用 `<type>: <简短描述>`。
- 推荐类型：`feat`、`fix`、`refactor`、`test`、`docs`、`ci`、`chore`。
- 描述使用祈使语气，简洁说明变更目的。
- 一次提交聚焦一个逻辑变更，避免混合无关格式化。
- 不提交密钥、签名文件、构建产物、本地配置和 IDE 工作区文件。
- 未经用户明确要求，不执行 destructive Git 操作。

## Kotlin 编码规范

- 遵循 Kotlin 官方代码风格和项目 `.editorconfig`。
- 使用不可变数据优先，限制可变状态的作用域。
- 公共 API 明确输入、输出和生命周期语义。
- 避免不必要的 nullable、强制解包和隐式副作用。
- 业务逻辑不放入资源层或 Composable 的渲染代码中。
- 新增异步逻辑必须明确协程作用域、取消和错误处理。

## Compose 编码规范

- Composable 保持单一职责，UI 状态通过参数传入，事件通过回调向上通知。
- 可复用 UI 组件不直接持有业务依赖或发起网络请求。
- 使用 `Modifier` 作为可组合组件的最后一个参数，并合理提供默认值。
- 使用 `stringResource` 管理用户可见文本，避免硬编码文案。
- 为交互元素提供明确的 content description 或可访问语义。
- Wear OS 列表优先使用适配 Wear 的组件和滚动容器。

## Material Design 3 规范

- 优先使用 Wear OS Material 3 组件、主题、颜色、排版和形状。
- 颜色必须满足前景/背景对比度和深色主题可读性要求。
- 交互状态应覆盖默认、按下、聚焦、禁用和错误状态（适用时）。
- 控件尺寸、间距和布局必须适配圆形屏幕及不同字体缩放。
- 不为了视觉效果绕过组件的语义、可访问性或系统返回手势。

## Repository 架构规范

- 当前仓库只有一个 Android `app` 模块；新增模块前必须有明确需求和方案确认。
- 当前工程使用 Compose for Wear OS；后续代码应优先保持该技术路线。
- 包名、namespace、applicationId 的调整属于高影响变更，必须单独评估和确认。
- UI、状态、数据访问和基础设施应保持职责分离；只有在业务需求出现后再引入对应层级。
- 不为尚不存在的业务过早创建复杂抽象。

## ViewModel 规范

- 当前项目尚未使用 ViewModel。
- 只有当页面出现跨重组、配置变化、异步任务或多状态协调需求时才引入 ViewModel。
- ViewModel 负责状态和事件协调，不负责直接渲染 UI。
- UI 通过不可变 State 订阅 ViewModel，并把用户事件传回 ViewModel。
- ViewModel 不持有 Activity、Composable 或 View 引用。

## API 规范

- 当前项目没有已确认的远程 API 或本地业务 API。
- 新增 API 前必须明确 endpoint、方法、认证、请求/响应模型、错误模型、超时和重试策略。
- API 契约必须先记录在 `docs/API.md`，再实现代码。
- 不在源码中硬编码密钥、令牌或环境相关地址。
- 网络调用必须有加载、成功、空结果、错误和取消语义（适用时）。

## 最小修改原则

- 只修改用户明确要求或完成任务必需的文件。
- 不顺便改名、升级依赖、重排目录、格式化整个项目或替换设计。
- 对现有用户改动保持尊重，不覆盖、不丢失、不擅自回退。
- 每次提交前检查 diff，确认变更与任务一一对应。

## 禁止猜测

- 不确定的业务名称、用户流程、数据格式、API、权限、发布策略统一标记“【待确认】”。
- 不以模板中的示例文案推断产品需求。
- 不以技术上“通常如此”替代项目实际证据。

## 禁止需求扩散

- 不把建议自动变成实现。
- 不因发现潜在问题而修改无关模块。
- 发现风险时记录到文档并在汇报中提出，不未经授权处理。

## 文档同步要求

- 每完成一个开发任务，更新 `DEVELOPMENT_PLAN.md` 和 `CHANGELOG.md`。
- 架构变化更新 `ARCHITECTURE.md`，接口变化更新 `API.md`。
- UI 规则变化更新 `UI_GUIDELINES.md`，测试策略变化更新 `TEST_PLAN.md`。
- 重要技术取舍更新 `DECISIONS.md`。

## 多 Agent 协作规范

- 所有 Agent 开始任务前必须阅读 `AGENTS.md` 和 `docs/`。
- 主 Agent 负责范围确认、方案整合、文档同步和最终验证。
- 子 Agent 必须明确任务边界，不得修改边界外文件。
- 不同 Agent 不应同时修改同一文件；发生冲突时由主 Agent 统一处理。
- 子 Agent 的结论必须标明依据、未验证项和待确认事项。
